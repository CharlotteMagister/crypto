    FORS

    FORS is the abbreviation for "Forest of random subsets". FORS is a "few-times signature scheme". It's an improvement from HORST, which is a variant of HORS. For the safety of FORS it is It is important that the call parameter be the output of a hash function is.

    Parameters used

    • n: defines the length of the secret and public Key, number of bytes in a signature element.
    • k: number of secret key groups, trees and indexes which are calculated from the input string.
    • t: number of elements per secret key group, number the leaves per hash tree and the upper limit of the calculated Index value. The parameter must be a multiple of 2.

    FORS uses the parameters k and t (t = 2 a). It signs the Strings of length k * a.

    The secret key consists of k * t random strings the byte length n. The keys are in k groups each with t Grouped elements. The character strings become pseudorandom from Generated secret key SK.seed.

    A public FORS key is a hash of length n is calculated from a hash of the root nodes of the k binary trees.

    Each of the binary trees has a height a and is used by t to authenticate secret keys of the k groups.

    FORS signature

    A FORS signature consists of k secret key elements and the authentication path.

    To compute a FORS signature, the message hash (md, message digest) is divided into k bit strings of length a. This is too Note that a is the number of bits! The bit strings then become interpreted as a number between 0 and t-1.
    To a secret To obtain keys, the sets are sequentially selected and the sets become the secret key with the calculated index selected.

    The authentication path is calculated by always using the opposite node is selected.

    FORS TreeHash

    The parameters used are:

    • SK.seed, the secret main initialization vector
    • s, the start index
    • z, the height of the destination node
    • PK.seed, the main public initialization vector
    • adrs, the address of the tree

    FORS-TreeHash describes the calculation of the nodes with the length n. The function works much like the TreeHash function of the Merkle tree. It is assumed that 2% 2 z = 0, since otherwise the Treehash algorithm would calculate nodes which are not in the tree.

    FORS signature generation

    A FORS signature consists of k parts, with each part being a secret Key is n bytes and the associated authentication path with ld (t) * n bytes.
    You can unlock the key generation split several steps:

    1. For each signature element:
      1. Calculate the next index (t bits from the message hash are interpreted as a number)
      2. Choose a secret key
      3. Calculate the authentication path (call the TreeHash function)
      4. Append the key and path to the FORS signature
    2. Return the entire FORS signature

    A signature has the following scheme:

    Private key value (tree 0) (n bytes)
    Authentication path (tree 0) (log t * n bytes)
    . . .
    Private key value (tree k-1) (n bytes)
    Authentication path (tree k-1) (log t*n bytes)

    WOTS + one-time signatures

    • w: The winternitz parameter is an element of the list {4, 16, 256}. A larger value means smaller signatures, however slower operations.

    The number of elements in a WOTS + signature, a public one Key and a private key can be expressed by the following formula calculate: len = len 1 + len 2 . len 1 = ceil (n / log (w))
    len 2 = floor (ld (len 1 (w-1)) / ld (w)) + 1

    WOTS + chain function

    The WOTS + chain function calculates an iteration of the Hash function F over the n-byte input string under the Using ADRS and PK.seed.

    The parameters used are:

    • X, the input string
    • i, the start index
    • s, the number of steps
    • PK.seed, the public initialization vector
    • ADRS, seven fields each with 32-bit words around the address to be able to contain the chain

    Run:

    1. Check if S == 0, if yes, return X
    2. Check if (i + s)> (w-1), if yes, return NULL
    3. Recursive call of the function, with the same parameters, except s-1. Save the return value in tmp.
    4. set the hash address to i + s-1
    5. Return the hash of (PK.seed, ADRS, tmp);

    WOTS + Sign

Then the base of the numbers is changed to w (length len 1 ). A checksum over M is calculated and also based on w converted (length len 2 ).     Each the numbers to the base w are used around a node of one to use another hash chain. The signature arises from the Joining the different nodes. The parameters are the Message M, SK.seed, PK.seed, ADRS.

    Run

    2. Calculate checksum
    3. Convert the checksum to the base w

      2. Calling the chain function with the hash of SK.seed and ADRS, 0, message at current position, PK.seed and ADRS. Append result to the current position of the signature.

    XMSS

    Treehash

    For the calculation of the internal nodes of the Merkle tree, which n bytes long, the treehash function is used. The used parameters are:

    • SK.seed
    • PK.seed
    • s, the start index. Starting at the leaf of the tree, and starts with 0.
    • z, the height of the destination node
    • ADRS, the address of the tree

    The algorithm returns the root of a tree with height z back, where the leaf, which is on the left, the public WOTS + key with the index s.

    A prerequisite for this algorithm is that s% 2 z = 0 otherwise, the algorithm would become non-existent nodes to calculate. The algorithm has the following sequence:

    1. Check if s% 2 z = 0, otherwise abort
    2. For each index i, until 2 z :
      1. Set the address of the WOTS hash
      2. Calculate a node by computing a public one WOTS key
      3. Set tree height and tree index
      4. For each node on the stack that has the same height as the calculated node:
        1. Set the index of the tree
        2. Calculate the new node by hovering the Concatenation of the node on the stack with the previously calculated Node
        3. Set the height of the tree
      5. Add stack to recalculated node
    3. Return the node

    The SPHINCS + Hypertree

    The SPHINCS + Hypertree is a variant of XMSS MT . It is a certification tree of XMSS instances. In this tree will always the overlying node uses the underlying node to sign. Each node, except the lowest level, is the public key of the XMSS tree. The lowest level (the Leaves of the hypertree) are used around the actual message to sign. These nodes are the public keys of FORS.

    Hypertree Verification

    A signature verification consists of the calls to XMSS, the returns the secret key from a signature, and a Comparison with a given value. The process is up to the last Step equal to signing:

    1. Sign the first node (root)

    3. If the last node is equal to the public key of the Hypertrees, then the signature is verified correctly.
